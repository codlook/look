LOOK nedir?

LOOK, web geliştirmeyi mümkün olduğunca doğrudan tutmak için tasarlanmış bir dildir. Route tanımla, veritabanına bağlan, JSON döndür — framework kurmadan, konfigürasyon dosyaları yazmadan.

Dil C++23 ile yazılmıştır ve Apache/XAMPP ortamına kopyalanıp çalıştırılabilir. FastCGI modunda sunucu başladığında scripti bir kez yükler; sonraki her istek hazır runtime üzerinden işlenir.

# Bir LOOK uygulaması bu kadar basit başlar

$conn = db::connect("mysql://root:@127.0.0.1/blog")
route("GET", "/yazilar", function() use ($conn) {
    $yazilar = db::query($conn, "SELECT id, baslik, tarih FROM yazilar ORDER BY tarih DESC", [])
    print(json::encode(["ok" => true, "data" => $yazilar]))
})
route("GET", "/yazilar/{id}", function($id) use ($conn) {
    $rows = db::query($conn, "SELECT * FROM yazilar WHERE id=?", [$id])
    if (count($rows) == 0) {
        response::status(404)
        print(json::encode(["ok" => false, "hata" => "Bulunamadı"]))
        return
    }
    print(json::encode(["ok" => true, "data" => $rows[0]]))
})
route("404", function() {
    response::status(404)
    print(json::encode(["ok" => false, "hata" => "Endpoint bulunamadı"]))
})

Başlarken

LOOK'u kurup ilk uygulamayı çalıştırmak için 5 adım — framework yok, konfigürasyon dosyası yok.

⊞ Windows (XAMPP)

  1. GitHub Releases'tan look-windows-x64.zip indir
  2. look-fcgi.exe ve look-cgi.exe dosyalarını C:\xampp\cgi-bin\ klasörüne kopyala
  3. C:\xampp\htdocs\.htaccess dosyasına ekle: Options +ExecCGI
  4. C:\xampp\htdocs\index.lk dosyası oluştur (aşağıdaki örnek)
  5. look-fcgi'yi başlat: look-fcgi.exe --port 9000
  6. Apache'yi başlat, http://localhost/ aç

Detaylı rehber: xampp/setup.md

⊞ Linux (Ubuntu / AlmaLinux)

  1. GitHub Releases'tan platforma uygun binary indir
  2. Binary'yi kopyala ve execute bit ver:
    cp look-fcgi /opt/look/ && chmod +x /opt/look/look-fcgi
  3. Uygulama dizininde index.lk dosyası oluştur
  4. systemd servisi yaz (bkz. FastCGI bölümü)
  5. Apache .htaccess rewrite kurallarını ekle (bkz. Apache bölümü)
  6. systemctl start look-fcgi → curl http://localhost/

Detaylı rehber: docs/ubuntu-deployment.md

Tasarım Kararları

LOOK bazı şeyleri bilinçli olarak dışarıda bırakır. Bunlar kısıtlama değil, tasarım kararıdır.

Routing dile gömülü

Ayrı bir router kütüphanesi yok. route() built-in — dosyayı çalıştırdığında routelar otomatik kayıt olur, script sonu dispatch eder.

Global değişken yok

HTTP verisine request::get(), request::post(), request::json() ile erişilir. Session için session::, cookie için cookie::. Her şey açık, izlenebilir.

Closure capture açık

Dış değişkene erişmek için use ($conn) yazmak zorundasın. Hangi verinin nereye gittiği her zaman görünür.

OOP yok — kasıtlı

LOOK'ta class, interface veya trait yok. Web uygulamalarının büyük çoğunluğu fonksiyon + veri yapısı ile yazılabilir. İleride OOP gerektiren senaryolar için use orm; modülü planlanıyor (Phase 10).

Karma array yok

Sayısal dizi ve ilişkisel dizi ayrı kavramlardır, birbirine karışmaz. Veri yapısı netliği tercih edilmiştir.

function auth_required() {
    $header = request::header("Authorization") ?? ""
    if (string::starts_with($header, "Bearer ")) {
        $token = string::substr($header, 7)
    } else {
        $token = request::header("X-Token") ?? ""
    }
    $payload = jwt_verify($token, env("JWT_SECRET", ""))
    if ($payload == null) {
        response::status(401)
        print(json::encode(["error" => "Geçersiz token"]))
        return null
    }
    return $payload
}

FastCGI Warm Start

LOOK'un en önemli performans özelliği. FastCGI modunda index.lk yalnızca bir kez yüklenir, parse edilir ve yorumlanır.

Nasıl Çalışır

# look-fcgi.exe --port 9000

İlk istek gelir → setup fazı:
  1. index.lk parse edilir → AST (WarmApp::program'da kalır)
  2. set_setup_mode(true) → interpret()
     - route() çağrıları → route_registry_'e kaydedilir

Sonraki HER istek → dispatch fazı:
  1. make_web_context(req) → REQUEST_URI, METHOD, headers, cookies
  2. dispatch_routes() → pattern match → callback($param...)
  3. $conn hazır → yeni DB bağlantısı açılmaz

WarmApp::program — AST process boyunca hayatta kalmalıdır. Closure'lar AST node'larına pointer tutar.

Concurrent Runtime (Phase 12)

look-fcgi tam eş zamanlı çalışır. Her HTTP isteği ayrı bir worker thread'de işlenir; LOOK kodu hiç değişmez. Daha ileri seviye eş zamanlılık için parallel() + channel() kullanılır.

Başlatma

# Windows XAMPP — 4 worker thread
Start-Process "C:\xampp\cgi-bin\look-fcgi.exe" -ArgumentList "--port 9000 --workers 4" -WindowStyle Hidden

# Linux systemd
ExecStart=/opt/look/look-fcgi --port 9000 --workers 8

# workers belirtilmezse → hardware_concurrency() (en az 2)

Mimari

look-fcgi --port 9000 --workers N

Her istek → ThreadPool'dan bir worker thread:
  1. shared_mutex (dispatch süresi boyunca okunur kilit)
     hot reload gerekirse → yazma kilidi → run_setup() → okuma kilidine dön
  2. make_dispatch_copy()   — per-request hafif interpreter kopyası
  3. acquire_thread_connections()  — her pool'dan 1 bağlantı ödünç alınır
  4. dispatch_routes() / interpret()  — tamamen izole, thread-safe
  5. release_thread_connections()  — bağlantılar pool'a geri verilir
  6. HTTP yanıtı yazılır

# Hub channel — top-level (setup fazında yaratılır, tüm WS route'larına geçilir)
$hub = channel()
route("WS", "/chat", function($ws) use ($hub) {

    # Hub'dan mesaj alıp bu client'a ileten goroutine
    parallel(function() use ($ws, $hub) {
        while (true) {
            $msg = receive($hub)
            if ($msg == null) { break }   # hub kapandı
            ws::send($ws, $msg)
        }
    })
    # Mesaj gelince hub'a yaz, tüm clientlara yayınla
    ws::on($ws, "message", function($data) use ($hub) {
        $parsed = json::decode($data)
        $out    = json::encode(["from" => request::ip(), "msg" => $parsed["msg"]])
        send($hub, $out)
        ws::broadcast($out)
    })
    ws::on($ws, "close", function() {
        log::info("WS bağlantı kapandı")
    })
})

        ])
        $ok = sse::send($sse, $veri, "tick")
        if (!$ok) { timer::cancel($tick) }  # bağlantı kesildi
    })
    # 30 saniyede bir keepalive ping
    $ping = timer::every(30000, function() use ($sse) {
        sse::send($sse, "keepalive", "ping")
    })
    # Bağlantı kapanınca timer'ları iptal et
    sse::on($sse, "close", function() use ($tick, $ping) {
        timer::cancel($tick)
        timer::cancel($ping)
        log::info("SSE bağlantı kapandı")
    })
})

jobs::

SQLite tabanlı, kalıcı, thread-safe iş kuyruğu. Uygulama yeniden başlasa da işler kaybolmaz. Delayed jobs, retry mekanizması, dead letter queue ve worker loop dahil. use jobs; ile yüklenir.

Durum Yaşam Döngüsü

pending → processing → done
                    ↘ failed  (retry_count ≥ max_retries)
     fail() → pending   (retry_count < max_retries — otomatik yeniden deneme)
     retry() → pending  (manuel, sadece failed'dan)

API

FonksiyonAçıklama
jobs::push($queue, $payload [, $max_retries=3 [, $delay_sec=0]])İş ekle → id döner. delay_sec > 0 ise delayed job — şu andan N saniye sonra hazır olur
jobs::next($queue)Sonraki hazır pending işi al → assoc | null. Delayed joblar run_after geçene kadar görünmez
jobs::done($id)İşi tamamlandı işaretle → processing → done
jobs::fail($id)Hata — retry_count < max_retries ise pending'e, aşıldıysa failed'a geçer
jobs::stats($queue)→ {pending, processing, done, failed} sayıları
jobs::list($queue, $status [, $limit=100])İşleri listele → array
jobs::failed($queue [, $limit=100])Dead letter queue — jobs::list($q, "failed") kısayolu
jobs::retry($id)failed → pending (manuel yeniden deneme)
jobs::purge($queue, $status)Belirtilen durumdaki işleri sil → silinen sayısı
jobs::worker($queue, $fn)Kuyruğa handler kaydet. $fn($job) → true (done) / false (fail)
jobs::run([$interval_ms=5000])Worker döngüsü. interval_ms > 0 → sonsuz döngü (ayrı process). interval_ms = 0 → tek tur, döner (timer ile göm)

Dördüncü parametre delay_seconds — iş şu andan N saniye sonra hazır olur. jobs::next() hazır olmayanları görmez.

Sıkça Sorulan Sorular

Yazılımcıların LOOK'u ilk gördüklerinde sordukları teknik sorular ve yanıtları.

Performans

Tree-walk interpreter yavaş değil mi? PHP bile JIT kullanıyor.

"Tree-walk = yavaş" ezberi CGI modeli için doğrudur — her istekte parse + interpret. LOOK'ta FastCGI Warm Start sayesinde kod bir kez parse edilir, AST bellekte kalır. Her istek sadece dispatch → callback çalıştırır; parse maliyeti sıfır. Bu model PHP'nin CGI'ına değil, Node.js veya Go'nun net/http handler yapısına yakındır.

Dahası, web uygulamalarında darboğaz interpreter değildir: yanıt süresinin %70–80'i DB sorgusu, %10–15'i ağ/I/O'dur. Interpreter payı %5–10. LOOK, asıl darboğazı (DB) C++ bağlantı havuzuyla çözer. Phase 12 ile ThreadPool + ConnPool eklendi — N worker thread eş zamanlı çalışır. Saha kanıtı: 72 saatte 700k+ istek, 134 ms sabit ortalama yanıt, 0 CPU kiliti.

3.5 RPS çok düşük. Gerçek yük testinde ne olur?

72 saatlik test bir kararlılık testidir — RPS yüksekliği değil, "PID değişir mi, bellek sızar mı, crash olur mu" sorularını yanıtlar. Sıralı curl döngüsü kasıtlı olarak tek thread çalışır. Gerçek yük testi için wrk ile ölçülen değerler: DB'li 418 RPS, DB'siz 2.150 RPS (CGI'ya kıyasla 26×).

Güvenlik

String interpolation'da kod çalıştırılabiliyor. DB'den gelen veri eval edilir mi? (Injection riski)

Hayır. String interpolation {...} yalnızca kaynak koddaki string literal'larda, parse zamanında çalışır. Runtime'da bir değişkenin içindeki {...} düz metin olarak kalır, ikinci kez evaluate edilmez. 

# DB'den gelen ham veri
$db_value = file::read("/tmp/test.txt")  # içerik: {1+1}

print($db_value)  # çıktı: {1+1}   ← evaluate edilmedi
$s = "Deger: " . $db_value  # çıktı: Deger: {1+1}
$s2 = "Deger: {$db_value}"  # çıktı: Deger: {1+1}   ← çift tırnak da evaluate etmedi

DB'den, dosyadan veya kullanıcı girdisinden gelen hiçbir string bu süzgeçten geçemez. Code Injection / RCE teorik olarak imkânsızdır.

Dokümantasyonda "dangling pointer" uyarısı var. Sunucu çökebilir mi?

O uyarı dili kullanan yazılımcıya değil, çekirdek geliştiricilerine yönelik bir iç mimari kuraldır (WarmApp::setup_out'un local değil WarmApp üyesi olması gerektiği). LOOK üzerinde kod yazan geliştirici pointer'a dokunmaz; bellek yönetimi tamamen runtime tarafından yürütülür. Kanıt: 72 saat, 700k+ istek, PID hiç değişmedi.

Mimari

FastCGI'da $conn global kalıyor. MySQL wait_timeout'ta bağlantı koparsa uygulama patlar mı?

LOOK bu senaryoyu db_check($conn) kalıbı + otomatik reconnect ile çözer. MySQL recovery testinde MariaDB kasıtlı durduruldu: MySQL yeniden başlayınca look-fcgi hot reload tetiklenmeden kendi kendine reconnect etti (Adım 3 → 200). db_check() ise bağlantı yokken 503 döndürür; uygulama hiç çökmez.

OOP yok. Büyük projeler spagetti koda dönmez mi?

LOOK'ta class yoktur — kasıtlı olarak. Kod organizasyonu use modül_adı; ile dosya bazlı modüllere ve struct ile veri gruplamaya dayanır. Bu, Go'nun struct + package yaklaşımının aynısıdır. OOP'un gereksiz hale getirdiği inheritance zincirlerini, interface boilerplate'ini taşımaz. Kurumsal büyüklükteki QR Menu uygulaması (29 route, 50k+ ürün) bu modelle yazılmış ve production'da çalışıyor.

XAMPP/Apache modeli eski. Docker ve Kubernetes'te çalışır mı?

LOOK, Apache veya XAMPP'a bağımlı değildir. look-fcgi kendi TCP soket sunucusunu barındırır (port 9000). Tek satır Dockerfile ile paketlenip Kubernetes üzerinde çalışır. XAMPP, geliştirici bilgisayarında kurulum kolaylığı için bir seçenektir — zorunluluk değil. Linux binary'leri ayrıca derlenir: Ubuntu 24.04 ve AlmaLinux 8 için Docker build image'ları hazırdır.

Package manager yok. Üçüncü parti kütüphane nasıl eklenecek?

En çok ihtiyaç duyulan modüller (MySQL, SQLite, PostgreSQL, auth, validator, date, file, upload) harici bağımlılık olmadan çekirdeğe gömülüdür. use math; gibi bir satırla aktif olur. Topluluk paketleri için v0.26.0'da look install eklendi: look install github.com/user/repo komutu GitHub'dan zipball indirir, pkg/ klasörüne çıkartır ve look.lock ile kilitler. Registry yok, config yok.

Dil Tasarımı

$ işareti neden var? Modern dillerde yok.

Doğru bir eleştiri. $ prefixi string interpolation'da değişken ile ifadeyi ayırt etmek için v0.11.0'da tutuldu. v1.0 hedefinde bu tartışma açıktır; lexer sıfırdan C++23 ile yazıldığı için teknik engel yoktur.

use ($conn) hantallığı neden var? Lexical scope yok mu?

FastCGI warm start mimarisinde global scope setup fazında çalışır; closure'ların bu değişkenlere otomatik erişimi veri izolasyonu açısından risklidir. use ($conn) kasıtlı olarak bağımlılıkları görünür kılar — hangi route'un neye bağımlı olduğu kod okunurken anında anlaşılır. Lexical scoping tartışması v1.0 yol haritasında açık bir maddedir.

Karma array neden yok? JSON parse ederken sorun çıkmaz mı?

JSON, LOOK'ta assoc array olarak gelir: $data = json::decode($body); Tüm key'ler string, tüm value'lar LOOK tiplerine (int, float, string, bool, null, array, assoc) map edilir. JSON içindeki iç içe diziler de aynı şekilde çözümlenir. Karma array olmadığı için bellek düzeni öngörülüdür ve CPU cache dostu'dur.

LOOK'ta struct var mı?

Evet — Phase 11'de eklendi (v0.12.0). Go stili, inheritance yok, method yok, sadece veri: 

struct Kullanici {
    ad
    yas
    email
}

$u = Kullanici{ad: "Ali", yas: 30, email: "ali@example.com"}
print($u.ad)  # Ali
$u.ad = "Mehmet"  # güncelleme

# {"ad":"Mehmet","yas":30,"email":"ali@example.com"}

OOP (class/inheritance) kasıtlı olarak yoktur. Davranış ayrı fonksiyonlarla eklenir. Detaylar: struct bölümü.

LOOK eş zamanlı bir dil mi? parallel() ne zaman kullanılır?

LOOK iki seviyede eş zamanlıdır. Runtime seviyesinde: look-fcgi her HTTP isteğini ayrı bir worker thread'de otomatik olarak çalıştırır — geliştirici bunu görmez, kod değişmez. Dil seviyesinde: parallel() + channel() ile tek bir request içinde birden fazla işi paralel yapılabilir (örneğin 3 DB sorgusu eş zamanlı). 

# Sıradan route — kod değişmez, runtime paralel çalıştırır
route("GET", "/stats", function() use ($conn) {
    # Tek thread içinde sıralı
    $u = db::query($conn, "SELECT count(*) as n FROM urunler", [])[0]["n"]
    $k = db::query($conn, "SELECT count(*) as n FROM kategoriler", [])[0]["n"]
    print(json::encode(["urunler" => $u, "kategoriler" => $k]))
})
# parallel() ile — iki sorgu eş zamanlı çalışır
route("GET", "/stats-fast", function() use ($conn) {
    $r = channel(2)
    parallel(function() use ($r, $conn) {
        send($r, db::query($conn, "SELECT count(*) as n FROM urunler", [])[0]["n"])
    })
    parallel(function() use ($r, $conn) {
        send($r, db::query($conn, "SELECT count(*) as n FROM kategoriler", [])[0]["n"])
    })
    $u = receive($r) $k = receive($r)
    print(json::encode(["urunler" => $u, "kategoriler" => $k]))
})

route("WS", "/ws", function($ws) use ($hub) {

    # Hub'dan mesaj al → bu istemciye gönder
    parallel(function() use ($ws, $hub) {
        while (true) {
            $msg = receive($hub)
            if ($msg == null) { break }
            ws::send($ws, $msg)
        }
    })

    # Mesaj gelince hub'a yayınla
    ws::on($ws, "message", function($data) use ($hub) {
        $parsed = json::decode($data)
        $paket  = json::encode([
            "from"  => request::ip(),
            "mesaj" => $parsed["mesaj"]
        ])
        send($hub, $paket)
    })

    # 30 saniyede bir keepalive ping
    $ping = timer::every(30000, function() use ($ws) {
        ws::send($ws, json::encode(["ping" => true]))
    })

    ws::on($ws, "close", function() use ($ping) {
        timer::cancel($ping)
        log::info("Bağlantı kapandı. Aktif: " . ws::clients())
    })
})

route("GET", "/durum", function() {
    print(json::encode(["ws_clients" => ws::clients()]))
})

# Upload işle
route("POST", "/yukle", function() use ($conn) {
    $baslik = request::post("baslik") ?? ""

    # request::file() magic byte + MIME + boyut kontrolü yapar
    try {
        $dosya = request::file("dosya", [
            "max_size"   => 5242880,           # 5 MB
            "allow_mime" => [
                "image/jpeg",
                "image/png",
                "image/webp"
            ]
        ])
    } catch ($e) {
        response::status(400)
        print(json::encode(["ok" => false, "hata" => $e]))
        return
    }

    if ($dosya == null) {
        print(json::encode(["ok" => false, "hata" => "Dosya seçilmedi"]))
        return
    }

    # file::store() → web root dışına güvenli kaydeder, SHA-256 adı verir
    $kayit  = file::store($dosya, "resimler")
    $url    = "/uploads/resimler/" . $kayit["name"]

    db::exec($conn,
        "INSERT INTO dosyalar (baslik, url, boyut, mime) VALUES (?,?,?,?)",
        [$baslik, $url, $dosya["size"], $dosya["mime"]])

    print(json::encode([
        "ok"    => true,
        "url"   => $url,
        "boyut" => $dosya["size"],
        "mime"  => $dosya["mime"]
    ]))
})

Magic byte kontrolüUzantıya değil dosya içeriğine bakarak gerçek tipi doğrular. .php uzantısını .jpg ile gizleme girişimlerini engeller.
